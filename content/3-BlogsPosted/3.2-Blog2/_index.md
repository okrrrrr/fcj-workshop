# S3 PRESIGNED URL — SECURE FILE UPLOADS WITHOUT GOING THROUGH YOUR SERVER


## The Problem


Our team's project is a malware analysis platform — users need to upload suspect files for the AI engine to scan. Initially, the upload flow we designed was pretty straightforward:


```
Client → API Gateway → Lambda → write file to S3
```


Worked perfectly with small files of a few kilobytes. But when we tested with a 15MB file — **boom**, Lambda returned `413 Payload Too Large`. Turns out API Gateway has a maximum payload limit of **10MB**, and Lambda's synchronous request body limit is **6MB**.


After some research, we found that virtually no production system on AWS uploads files through Lambda. Instead, they use a technique called **Presigned URL**.


## What is a Presigned URL?


A Presigned URL is a temporary URL generated by the AWS SDK that allows anyone who has it to perform a specific operation on S3 (e.g., upload or download) within a limited timeframe, **without needing AWS credentials**.


In other words: Lambda creates a "time-limited entry pass" for the client. The client takes that pass and uploads directly to S3, bypassing Lambda entirely.


## How It Works


![Presigned URL Flow](/fcj-workshop/images/3-Blog/blog2_presigned_url.png)


Here's how the upload flow works in our system:


1. **Client calls the API** `POST /upload` via API Gateway
2. **Lambda processes the request**: Validates the user via Cognito token, generates a unique object key (e.g., `uploads/{userId}/{timestamp}_{filename}`), then uses Boto3 to call `generate_presigned_url()` creating a Presigned PUT URL valid for 5 minutes
3. **Lambda returns the URL** to the client
4. **Client uploads directly** to S3 using an HTTP PUT request with the Presigned URL — the file goes straight from the browser to S3, no Lambda involved
5. **S3 Event Notification** fires when the new file lands in the bucket → pushes a message to SQS → Step Functions coordinates the analysis


### Lambda Code for Generating Presigned URL (Python)


```python
import boto3
import json
import uuid
from datetime import datetime


s3_client = boto3.client('s3')
BUCKET_NAME = 'malware-sandbox-raw-uploads'


def lambda_handler(event, context):
    # Get user info from Cognito
    user_id = event['requestContext']['authorizer']['claims']['sub']
    filename = json.loads(event['body'])['filename']
    
    # Generate unique object key
    timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
    object_key = f"uploads/{user_id}/{timestamp}_{filename}"
    
    # Generate Presigned URL (expires after 300 seconds = 5 minutes)
    presigned_url = s3_client.generate_presigned_url(
        'put_object',
        Params={
            'Bucket': BUCKET_NAME,
            'Key': object_key,
            'ContentType': 'application/octet-stream'
        },
        ExpiresIn=300
    )
    
    return {
        'statusCode': 200,
        'headers': {
            'Access-Control-Allow-Origin': '*'
        },
        'body': json.dumps({
            'upload_url': presigned_url,
            'object_key': object_key
        })
    }
```

## Mistakes We Made (and How We Fixed Them)


### Mistake 1: CORS — "No 'Access-Control-Allow-Origin' header"


This was the first error we hit when the client tried to PUT to S3. The browser blocked the request because the S3 bucket had no CORS configuration.


**Fix:** Go to S3 bucket → Permissions → CORS configuration:
```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["PUT", "GET"],
    "AllowedOrigins": ["https://your-domain.com"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3600
  }
]
```


### Mistake 2: Content-Type mismatch


When generating the Presigned URL, we set `ContentType: 'application/octet-stream'`. But the client either forgot to set the `Content-Type` header or set a different value → S3 rejected the request with `403 SignatureDoesNotMatch`.


**Fix:** The Content-Type in the Presigned URL params and in the client's request header **must match exactly**.


### Mistake 3: URL expired and we couldn't figure out why


We set `ExpiresIn=60` (1 minute) to be "safe". Result: large files on slow networks didn't finish uploading before expiration.


**Fix:** Increase `ExpiresIn` to 300 seconds (5 minutes) or 900 seconds (15 minutes) for large files. The URL is single-use so there's no abuse risk.


## Comparison: Upload via Lambda vs Presigned URL


| Criteria | Via Lambda | Presigned URL |
|---|---|---|
| File size limit | 6MB (sync) / 10MB (API GW) | **5GB** (single PUT) |
| Compute cost | Lambda duration charges | **Nearly zero** (only 1 call to generate URL) |
| Speed | Slower (through 2 proxy layers) | **Fast** (direct to S3) |
| Complexity | Simpler initially | Requires extra URL generation step |
| Best for | Small files < 5MB | **Files 1MB and above** |


## Lessons Learned


1. **Don't upload binary files through Lambda** unless they're tiny (< 1MB) — you hit size limits and waste compute money
2. **Presigned URL is the AWS best practice** for file uploads — virtually all production systems use it
3. **CORS is the biggest trap** — configure it before testing from a browser
4. **Content-Type must match** — this is the #1 cause of mysterious 403 errors
5. **Combine S3 Events + SQS** to create a complete asynchronous processing flow after upload
